# The big picture
**A free-rider** is a client in federated learning who wants the valuable global model but doesn't want to do the work. So instead of actually training on local data, they submit fake or recycled updates each round and still receive the shared model. 
**FareMark's core idea** is to force every client to stamp their own private, invisible signature ("watermark") into the model as a byproduct of genuinely training it. A real contributor's signature survives in the model and can be recovered later. A free-rider, who never trained, has no valid signature, so the server catches them.

# The method 
Runs across three phases: 
1. a pretraining step where the server hands out the rules, 
2. a training step where watermarks get embedded and free-riders get flagged, 
3. and a deployment step where the same watermarks double as copyright proof. 
The single hardest technical problem is that many clients are stamping signatures into one shared model at the same time, and their goals naturally collide. Most of Section IV is machinery to 
- (a) make these many watermarks coexist without conflict, 
- (b) make each watermark robust enough to survive averaging and tampering, and 
- (c) read them back out reliably.
The paper frames the **conflict question** explicitly: all clients are optimizing the same model but pulling it in different directions, so there's no single model that satisfies everyone. FareMark's answer is a server-assigned *"trigger class"* per client. Each client only cares about how the model behaves on one specific class of inputs (say, client 1 owns "dog," client 2 owns "cat"). Because each client operates on a different slice of the model's behavior, they stop stepping on each other. Conceptually, the decision space is carved into independent sub-regions, one per client.

## A. Watermark representation — reading a signature out of the softmax
Older watermarking methods hide the signature either in the weights (which means you need full white-box access to read it back) or as a backdoor (train the model to misclassify some special trigger image, which is fragile and gets erased by fine-tuning). FareMark avoids both. It reads the watermark out of the model's softmax probability vector on ordinary trigger-class inputs.

*Here's the mechanism.* The model outputs a probability vector $\hat{P} = [\hat{p}_1, \dots, \hat{p}_n]$ over the $n$ classes. To encode an $m$-bit watermark, you split this vector into $m$ groups and, for each group $k$, take a dot product with a secret pseudorandom direction (a row of a secret matrix $M$, which acts like a cryptographic key):
$$z^i_k = \sum_j \hat{p}^k_j , M_{i,k,j}$$
Then you read off one bit per group from the sign of that dot product: negative → 0, non-negative → 1 (Eq. 2). 
So the watermark isn't stored anywhere explicit; it's the pattern of signs you get when you project the model's outputs onto a set of secret directions. This is what *"box-free"* means: you don't need a specially crafted trigger image (any image of the trigger class works), and you don't need to open up the model.
The catch, and why the function $f$ exists. Softmax outputs are spiky. After normal training, one class gets something like 0.95 and the rest are near zero. The paper's worked example makes the problem concrete: if the probability vector is roughly $[0.01, 0.95, 0.04]$ and the secret direction is $[+1, -1, +1]$, the dot product is dominated entirely by that one big 0.95 term, so its sign in the secret key single-handedly decides the bit. You've lost the ability to choose what bit comes out. (The paper's text around Eqs. 3–6 is a little loose about which bit value it's illustrating, but the takeaway is solid: one dominant probability hijacks the projection.)
The fix is to pass each probability through a **flattening function $f$** before projecting, turning the spiky distribution into a smoother one so that several entries contribute meaningfully and you regain control over the sign. The paper offers three candidates: a power function with negative exponent $x^\alpha,\ \alpha<0$; a power function with fractional exponent $x^\alpha,\ 0<\alpha<1$; and a sinusoid $\sin(\alpha x)$, with $\alpha$ tuning how aggressively you flatten (Eqs. 7–9, illustrated in Fig. 6). The final extraction (Eq. 10) projects the flattened probabilities and adds a constraint that, after flattening, the largest single entry must account for less than half the total — a formal way of saying "no single class is allowed to dominate the readout."

## B. Client-side watermark embedding — making training also write the signature
Now, how does the watermark actually get into the model? Through the **loss function**. The server gives each client one trigger class; the others are "common classes." For common-class training images, the client just uses the normal classification loss. For trigger-class images, the client adds a second penalty that punishes the model whenever the secret signature can't be cleanly recovered from the output. So the total objective is:
$$L = L_{cl} + \lambda L_{wm}$$
where $L_{cl}$ is ordinary cross-entropy classification loss and $\lambda$ is a knob trading off "classify well" against "carry the watermark well." The watermark term $L_{wm}$ (Eq. 12) is just a binary cross-entropy: it feeds the projected value $\tilde{z}^i_k$ through a sigmoid and pushes it strongly positive when the target bit is 1 and strongly negative when it's 0. In plain terms, the model is trained to produce output probabilities on trigger-class images that, once projected onto the secret directions, land on the correct side of zero for every bit. The watermark is thus embedded as part of learning the task, not bolted on afterward.

## C. Memory-enhanced model updating — keeping the signature alive through averaging
This subsection fixes a problem created by federated averaging itself. In vanilla FL, every round the server averages all the local models into a global one, and each client then restarts from that global model. Two things go wrong for watermarks: (1) averaging many models together blurs out the delicate watermark patterns each client baked in, and (2) restarting from the global model each round throws away the client's own watermark-bearing tweaks.
FareMark's fix is a **modified local update rule** (Eq. 14). Instead of starting the new round purely from the global model, each client blends its own previous local model (which still carries its watermark) with the global model:
$$W^j_{i+1} = \beta\left(W^j_i + lr \cdot \tfrac{\partial L}{\partial W^g_i}\right) + (1-\beta)W^g_i$$
The hyperparameter $\beta$ controls the balance: high $\beta$ means "trust and preserve my own watermarked trajectory," low $\beta$ means "snap back toward the consensus model." This is the "memory" — the client remembers its own signature across rounds rather than letting aggregation wash it away. A nice side effect the authors point out: because the global model is a mixture of everyone's local models, looking at it gives each client a little indirect information about the others, which helps the separate watermarks settle into a compatible configuration even though clients never communicate directly. The cost they're honest about is that this anchoring slows convergence.

## D. Free-rider detection — the actual catching
Detection has three stages.
1. First, registration: before/after training, each client sends the server its secret key and its intended watermark message (optionally encrypted), and the server stores a record tying together the trigger class, the key, the message, and the client's identity.
2. Second, extraction: to check a client, the server takes $N_T$ images from that client's trigger class, queries the model with each, and averages the projections across all those samples before thresholding into bits (Eq. 15). Averaging over many images is what makes the readout robust — a single image can be noisy, but the consensus over many is stable.
3. Third, detection: the server compares the recovered bit-string to the one the client registered, using a normalized bit-error rate (Eq. 16) — essentially "what fraction of bits disagree." If the error is below a threshold $\eta$, the watermark is judged present and the client is a genuine contributor. The choice of $\eta$ matters because it trades off catching free-riders against falsely accusing honest clients. They set it statistically: measure the typical bit-error of legitimate clients over many rounds, then set $\eta = \mu + 3\sigma$ (the standard "3-sigma" rule), which keeps false positives very rare.
The subsection ends with the key argument for why a free-rider can't cheat the test. A clever free-rider might query the model with some random trigger-class image, run extraction with a random key, get whatever bits fall out, and register those. But without having actually trained, the model has no consistent watermark behavior across the whole trigger class — so the bits the free-rider extracted from their one image won't match the bits the server later extracts from different trigger-class images. The mismatch exposes them.

## E. Authentication of IPR — the same trick as copyright proof
Finally, the authors note the watermark does double duty. Once the model is trained and released, it might be stolen via model-extraction attacks or copied by an insider. Because each contributing client's signature is durably embedded in the model, the contributors (and the training agency) can prove ownership of a stolen model by extracting their registered watermarks from it. So the same mechanism that distinguishes contributors from free-riders during training also serves as a copyright stamp after deployment.
